Error Handling

Front-end level

The case starts with a user who makes a request, providing required and optional information.

If the provided data is not fit according to the form validations, it will not be able to submit and continue with the workflow.

When the user supplies with all the needed data and submits the enabled form, happens the following steps:

  • The Component calls a Service function.
  • The Service calls an API Service method.
  • The API makes an HTTP request to the server using the user-provided data.
  • That request could end in one of two possible results:
    • Good result:

      The outcome will return a known type response to the Service, which runs some service-level logic and return the molded response to the Component where the Service has started and given to the user a visual confirmation of the work.

    • Bad result:

      The API Service will catch the error and parse it to return only the provided error message to throw it to the Service, which will send it to the Component and give a visual response to the user.

Back-end level

When the front-end requests the server, it could end in one of two possible results:

  • Band result
    • The Middle-ware (an intermediate-level layer related to authentication) catches a mistake using its specific validations and returns an error response (with a unique error code and a friendly message for the user) to be handled by the front-end.
    • The business layer logic on the Route founds an error and returns it as mentioned the point before.
    • The model layer (coded in another part of the app to give shape to the data to be handled by the Route) finds an error trying making changes in the database and returns it as mentioned the point before
      • Not having included a required field.
      • Trying to use an already used email or username or some data with a length lower than the minimum required.
  • Good result
    • The request make all changes and returns a response to the front-end.
